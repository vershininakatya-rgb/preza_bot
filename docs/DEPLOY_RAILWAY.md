# Развёртывание бота на Railway (24/7)

Пошаговая инструкция для запуска Telegram-бота в облаке Railway без необходимости держать компьютер включённым.

---

## Автоматический деплой с GitHub

Когда репозиторий **подключён к Railway** (шаг 2 ниже), каждый **push в выбранную ветку** (обычно `main`) автоматически запускает новый деплой. Railway получает webhook от GitHub — отдельный GitHub Action для деплоя не нужен.

- Убедитесь, что в Railway в настройках сервиса указана нужная ветка (например, `main`).
- Опционально: в Railway можно включить **Wait for CI** — тогда деплой начнётся только после успешного прохождения GitHub Actions (см. `.github/workflows/ci.yml`).

---

## Что уже сделано в проекте

- Настроены файлы для Railway: `Procfile`, `runtime.txt`, `railway.toml`
- Бот готов к запуску из переменных окружения (без обязательного `.env` на сервере)
- В этой инструкции — только шаги, которые нужно выполнить вам

## Шаг 1. Регистрация в Railway

1. Откройте в браузере: **https://railway.app**
2. Нажмите **Login** (или **Start a New Project**).
3. Войдите через **GitHub** (удобнее всего, если код бота уже в GitHub).
4. Подтвердите доступ Railway к вашему GitHub при первом входе.

---

## Шаг 2. Создать новый проект

1. В личном кабинете Railway нажмите **New Project**.
2. Выберите **Deploy from GitHub repo**.
3. Если репозиторий с ботом не виден — нажмите **Configure GitHub App** и выдайте Railway доступ к нужному репозиторию (или ко всем).
4. Выберите репозиторий **preza_bot** (или как у вас называется проект).
5. Railway сам определит, что это Python-проект, и начнёт первый деплой. Дождитесь окончания сборки (может занять 2–5 минут).

---

## Шаг 3. Добавить переменные окружения

Без этих настроек бот не запустится. Переменные задаются в Railway, а не в коде.

1. В проекте Railway откройте **ваш сервис** (карточка с названием репозитория).
2. Перейдите на вкладку **Variables** (или **Variables** в боковом меню).
3. Нажмите **+ New Variable** или **Raw Editor** и добавьте переменные по одной.

### Обязательные переменные

| Переменная      | Где взять | Пример значения |
|-----------------|-----------|------------------|
| `BOT_TOKEN`     | Telegram: @BotFather → ваш бот → Copy Token | `8477985950:AAE8d4nTsEoN2WwVsVs-...` |

Без `BOT_TOKEN` бот не стартует.

### Важные для полной работы бота

| Переменная      | Назначение | Пример |
|-----------------|------------|--------|
| `LLM_API_KEY`   | Анализ и диаграммы (OpenAI) | `sk-proj-...` (из OpenAI API keys) |

### Запросы «Нужна помощь» — в ваш Telegram

| Переменная      | Назначение |
|-----------------|------------|
| `ADMIN_CHAT_ID` | Ваш числовой Telegram Chat ID — сюда бот пришлёт сообщение, когда пользователь нажмёт «Нужна помощь» и напишет текст. Узнать свой chat_id: написать боту [@userinfobot](https://t.me/userinfobot) в Telegram. |

### Логи мониторинга (канал/чат для логов)

| Переменная      | Назначение |
|-----------------|------------|
| `LOG_CHAT_ID`   | Chat ID канала или чата для сводных логов (одно сообщение на пользователя за сессию, после 5 мин бездействия). **Для каналов и групп указывайте с минусом**, например: `-5272065214`. Без минуса будет «Chat not found». Если не задан — можно задать, отправив боту `/start` в нужном чате/канале. |

### Опциональные

| Переменная           | Назначение |
|----------------------|------------|
| `DEBUG`              | `False` для продакшена |
| `LOG_LEVEL`          | `INFO` (по умолчанию) |
| `PYTHONUNBUFFERED`   | `1` — отключить буфер stdout; логи в Railway появятся сразу (в коде уже включён сброс буфера для логгера, эта переменная нужна для любого другого `print`) |

### БД: Supabase или PostgreSQL (RAG и сохранение данных бота)

Бот может сохранять в БД пользователей, аналитику проблем, диаграммы решений и обратную связь «Нужна помощь». Используется одна переменная **DATABASE_URL**.

**Вариант 1 — Supabase (рекомендуется):**

1. Создайте проект на [supabase.com](https://supabase.com), включите расширение **vector** (Project Settings → Database → Extensions).
2. Скопируйте connection string (URI) из Project Settings → Database.
3. В Railway → **Variables** добавьте **DATABASE_URL** со значением этого URI.
4. Один раз выполните инициализацию схемы (локально с `DATABASE_URL` из .env или через скрипт): `python -m scripts.init_supabase_schema`. Подробно: [docs/SUPABASE_SETUP.md](SUPABASE_SETUP.md).
5. При необходимости: **RAG_ENABLED** = `true`, **PERSIST_TO_DB** = `true` (по умолчанию запись в БД включена при наличии DATABASE_URL).

**Вариант 2 — PostgreSQL на Railway:**

- Добавьте плагин **PostgreSQL** в проект Railway — Railway подставит **DATABASE_URL** автоматически.
- Выполните один раз `python -m scripts.init_supabase_schema` (с DATABASE_URL из переменных) для создания таблиц.
- Вручную добавьте **RAG_ENABLED** = `true` при использовании RAG.

**Отключение сохранения в БД:** задайте **PERSIST_TO_DB** = `false`, если нужен только RAG без записи пользователей/аналитики/диаграмм/обратной связи.

После добавления переменных Railway перезапустит сервис сам.

## Шаг 4. Команда запуска (если в логах «No start command was found»)

В проекте есть файл **railpack.json** с командой `python main.py` — Railpack должен подхватить его. Если ошибка всё равно появляется:

1. В Railway откройте ваш сервис → вкладка **Settings**.
2. Найдите **Build & Deploy** → **Custom Start Command** (или **Start Command**).
3. Введите: `python main.py` и сохраните.
4. Сделайте **Redeploy** (пересборку).

У бота нет веб-сервера, только фоновый процесс; порт указывать не нужно.

---

## Шаг 5. Проверить, что бот работает

1. Во вкладке **Deployments** должен быть последний деплой со статусом **Success** (зелёная галочка).
2. Откройте вкладку **Logs** (или **View Logs**). В логах должно быть что-то вроде:
   - `Бот запущен...`
   - `Application started`
3. В Telegram откройте вашего бота и отправьте `/start`. Бот должен ответить.

Если в логах ошибка про `BOT_TOKEN` — проверьте переменную **Variables**. Если ошибка про порт или web — убедитесь, что в **Start Command** указано именно `python main.py` (без gunicorn и без привязки к порту).

## Автодеплой при пуше в GitHub

Чтобы Railway сам подхватывал изменения после каждого `git push`:

1. Откройте **Railway** → ваш проект → выберите **сервис** (карточка с ботом).
2. Перейдите в **Settings** (иконка шестерёнки или вкладка настроек).
3. Найдите блок **Source** или **Repository** / **GitHub**.
4. Убедитесь, что указан **ваш репозиторий** и ветка (обычно **main** или **master**).
5. Проверьте, что **не нажато** «Disconnect» — репозиторий должен быть **подключён**.
6. Если есть опция **Auto Deploy** или **Deploy on Push** — она должна быть **включена** (по умолчанию включена при подключении репо).

После этого каждый **push в выбранную ветку** в GitHub будет запускать новый деплой. Статус сборки смотрите во вкладке **Deployments**.

## Обновление бота (после изменений в коде)

1. Закоммитьте и запушьте изменения в GitHub в ту ветку, с которой связан сервис (чаще всего **main**).
2. Если автодеплой включён (см. выше), Railway сам начнёт новый деплой.
3. Статус — во вкладке **Deployments**, логи — в **Logs**.

Ручной деплой: в Railway откройте сервис → **Deployments** → **Deploy** / **Redeploy** (или Command Palette → «Deploy Latest Commit»).

## Если что-то пошло не так

1. **Логи** — вкладка **Logs** в вашем сервисе: там видно сообщения об ошибках при старте и работе бота.
2. **Переменные** — убедитесь, что **BOT_TOKEN** и остальные заданы без лишних пробелов и кавычек (значение вводится без кавычек, если иное не указано в подсказке Railway).
3. **Start Command** — должна быть ровно команда: `python main.py`.

---

## Почему деплой может не работать

| Симптом | Что проверить |
|--------|----------------|
| Сборка падает с «No start command» | В репозитории есть **railpack.json** и **railway.toml** с `startCommand: "python main.py"`. Либо в Railway → Settings → **Custom Start Command** вручную укажите `python main.py`. |
| Сборка успешна, бот не отвечает | Откройте **Logs** (раздел Deploy/Runtime). Если есть **409 Conflict** — запущен ещё один экземпляр с тем же BOT_TOKEN (например, локально). Остановите локальный бот: `make stop`. |
| Ошибка про BOT_TOKEN | В Railway → **Variables** добавлена переменная **BOT_TOKEN** (без кавычек в значении). После изменения Variables сервис перезапустится сам. |
| Деплой не стартует после push | Railway → Settings → **Source**: репозиторий подключён, выбрана ветка **main** (или ваша), нет кнопки Disconnect. При необходимости нажмите **Redeploy** вручную. |

Если опишете текст ошибки из логов (или скрин), можно подсказать точнее.

## Почему не приходят логи мониторинга в канал

При старте бот отправляет в канал логов тестовое сообщение: «✅ Логи мониторинга подключены...». Если его **нет**:

| Проверка | Действие |
|----------|----------|
| **LOG_CHAT_ID задан?** | В Railway → **Variables** добавьте **LOG_CHAT_ID** со значением chat_id канала **с минусом** (например `-5272065214`). Если указать без минуса (`5272065214`), в логах будет «Chat not found». Локально — в `.env`: `LOG_CHAT_ID=-5272065214`. |
| **Бот админ канала?** | В канале: Управление → Администраторы → добавьте бота → включите право **«Публикация сообщений»** (Post messages). |
| **В логах при старте** | Должна быть строка «Логи мониторинга будут отправляться в чат: -5272065214». Если видите «LOG_CHAT_ID не задан» — переменная не подхватилась (перезапустите сервис после добавления Variables). |

Если в логах ошибка **«Group migrated to supergroup. New chat id: -100...»** — чат был группой и стал супергруппой. Бот сам подхватит новый id и будет слать логи в новый чат. Чтобы не было лишней ошибки при каждом перезапуске, в Railway Variables задайте **LOG_CHAT_ID** равным этому новому id (например `-1003804112961`).

Если **тестовое сообщение приходит**, а сводки по пользователям — нет: сводка отправляется **через 5 минут** после последнего действия пользователя. Для проверки задайте в Variables или `.env`: **LOG_SESSION_TIMEOUT_SEC=60** — тогда сводка уйдёт через 1 минуту неактивности.
