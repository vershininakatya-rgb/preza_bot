# Анализ корневой причины проблем

**Дата:** 2026-02-22

---

## 1. Проявления проблемы (симптомы)

| Событие | Симптом | Реакция сегодня |
|---------|---------|-----------------|
| Запуск бота | 409 Conflict: "terminated by other getUpdates request; make sure that only one bot instance is running" | Объяснили: убить все процессы, запустить один |
| Перезапуск | Нужно вручную pkill, потом make run | Запустили pkill + make run из Cursor |
| Не работает бот | 401 Unauthorized (неверный токен) | Объяснили: обновить .env |
| Где запускать | Неочевидно | Объяснили: cd preza_bot && make run |

---

## 2. Что исправляли сегодня (и почему это не источник)

| Правка | Категория | Связь с корневой причиной |
|--------|-----------|---------------------------|
| Удаление `import asyncio` | Косметика | Нет |
| Упрощение `except (ValueError, Exception)` | Качество кода | Нет |
| Объединение сообщений в 0H | UX | Нет |
| Логирование в LLM fallback | Отладка | Частично — помогает видеть сбои, но не устраняет причину |
| Проверка placeholder LLM | Edge case | Нет |
| Обновление ARCHITECTURE | Документация | Нет |
| Создание Makefile | Инфраструктура | Частично — даёт `run`, но не решает конфликт экземпляров |

Все правки — точечные. Ни одна не устраняет причину повторяющихся сбоев.

---

## 3. Корневая причина

**Отсутствие операционного контракта и управления жизненным циклом процесса.**

Бот спроектирован как «запустил и забыл»:
- Не проверяет, не запущен ли уже другой экземпляр
- Не даёт понятной обратной связи при 409 (только сырой лог)
- Не предлагает действий («остановите другой экземпляр: …»)
- Нет явного способа остановить/перезапустить/проверить статус
- Конфигурация «как и где запускать» — неформальная, в голове разработчика

В результате:
- Cursor или другой терминал может оставить бота в фоне
- Пользователь запускает ещё один экземпляр → 409
- Ошибка приходит от Telegram API, а не от приложения
- Пользователь не понимает, что делать, кроме «убить всё и запустить заново»

---

## 4. Почему это системная проблема

1. **Telegram API** допускает только один активный `getUpdates` на токен. Это внешнее ограничение.
2. **Приложение** не учитывает это ограничение: не проверяет дубликаты, не обрабатывает 409 осмысленно.
3. **Инструменты** (Makefile) не управляют жизненным циклом: есть только `run`, нет `stop` / `restart` / `status`.
4. **Среда** (Cursor, несколько терминалов) увеличивает риск случайного запуска нескольких экземпляров.

Корень — в том, что приложение и инструменты не учитывают ограничение «один экземпляр на токен» и не помогают пользователю его соблюдать.

## 5. Предлагаемые пути устранения (на уровне источника)

### Вариант A: Обработка ошибок (минимальный)

**Суть:** Переводить 409 в понятное сообщение и подсказку по действиям.

**Что сделать:**
- В `error_handler`: при `Conflict` (409) логировать текст вида:  
  «Другой экземпляр бота уже запущен. Остановите его: `pkill -f "python run.py"` или `make stop`».
- Добавить в Makefile цель `stop` (pkill по `run.py`).

**Плюсы:** Мало изменений, сразу понятно, что делать.  
**Минусы:** Не предотвращает конфликт, только объясняет его.

---

### Вариант B: Управление жизненным циклом в Makefile

**Суть:** Сделать Makefile единой точкой входа для запуска/остановки/перезапуска.

**Что сделать:**
- `make stop` — остановить все процессы `run.py`.
- `make restart` — stop + run.
- `make status` — показать, запущен ли бот (например, через `pgrep`).
- `make run` — перед запуском проверять, не запущен ли уже бот; если да — выводить предупреждение и не запускать (или предложить `make restart`).

**Плюсы:** Явный контракт, меньше шансов на двойной запуск.  
**Минусы:** Не защищает от запуска `python run.py` в обход Makefile.

---

### Вариант C: Блокировка на уровне приложения (pidfile/flock)

**Суть:** Перед `run_polling` проверять, не занят ли «слот» для этого бота (через pidfile или file lock).

**Что сделать:**
- При старте: создать lock-файл (например, `.bot.lock` или pidfile).
- Если файл уже существует и процесс жив — выйти с сообщением: «Бот уже запущен (PID …). Остановите: make stop».
- При нормальном завершении — удалять lock.
- Обрабатывать SIGTERM/SIGINT для корректного снятия блокировки.

**Плюсы:** Защита на уровне приложения, не зависит от способа запуска.  
**Минусы:** Сложнее, нужно аккуратно обрабатывать аварийные завершения (чтобы не оставался «мёртвый» lock).

---

### Вариант D: Комбинированный (рекомендуемый)

**Суть:** Соединить A + B + упрощённый C.

1. **error_handler:** при 409 — понятное сообщение + подсказка `make stop` / `pkill`.
2. **Makefile:** цели `stop`, `restart`, `status`; в `run` — проверка через `pgrep` и предупреждение при уже запущенном боте.
3. **run.py / main.py:** опциональная проверка pidfile перед `run_polling` (можно включить через флаг или переменную окружения).

**Плюсы:** И профилактика (Makefile, опционально pidfile), и понятная реакция при 409.  
**Минусы:** Чуть больше кода и логики.

---

## 6. Что не является корневой причиной

- Отдельные баги в handlers (error_handler, update is None) — следствия, не источник повторяющихся сбоев.
- Токен 401 — разовая проблема конфигурации.
- Структура документации — не влияет на 409 и жизненный цикл.
- Логирование (DEBUG httpx и т.п.) — вопрос удобства отладки, не причина конфликтов.

---

## 7. Рекомендация

Начать с **варианта D**, поэтапно:

1. **Фаза 1:** error_handler для 409 + Makefile (`stop`, `restart`, `status`).
2. **Фаза 2:** проверка в `make run` перед запуском (pgrep).
3. **Фаза 3 (по желанию):** pidfile в приложении для жёсткой блокировки второго экземпляра.

Доработки не начинать до согласования плана.
