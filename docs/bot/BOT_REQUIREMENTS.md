# Требования к функционалу бота (на основе интервью с ЦА)

Документ описывает функциональные и нефункциональные требования к телеграм-боту для подготовки презентаций для C-level. Источник: интервью с деливери-менеджерами компаний 3000+ человек и описание продукта с учётом интервью.

---

## 1. Область и контекст

**Целевой пользователь:** деливери-менеджер, скрам-мастер, change-менеджер, агент изменений (компании 300+ человек).

**Цель продукта:**
- Помочь **составить вопросы для аналитиков и стейкхолдеров** и **общий план анализа проблемы** (кого опросить, какие данные собрать, какие гипотезы проверить).
- Помочь структурировать анализ проблемы (метрики, интервью, опросы).
- Снять страх «белого листа» при сборке презентации.
- Собрать презентацию в корпоративном шаблоне пользователя.

**Приоритет по интервью:** помощь в аналитике ценнее, чем только автоматическая верстка слайдов.

## 2. Онбординг и выбор сценария

### 2.1. Регистрация / первый запуск

| ID | Требование | Приоритет |
|----|------------|-----------|
| O-1 | При первом обращении к боту пользователь получает кратное приветствие и объяснение, что бот делает (1–2 предложения). | Must |
| O-1a | После приветствия бот предлагает **варианты, с чем помочь** (кнопки или меню). Пользователь выбирает один из вариантов: **помощь с аналитикой** (структурировать анализ проблемы, план исследования, метрики); **составить вопросы для интервью** (шаблоны вопросов по ролям, опросники); **подготовить презентацию** (от черновика до сборки в шаблоне); **полный цикл** (анализ + вопросы + презентация). Выбор ведёт в соответствующий сценарий. | Must |
| O-2 | Пользователь может в любой момент вызвать справку по командам и перейти в начало сценария. | Must |

### 2.2. Выбор роли и контекста

| ID | Требование | Приоритет |
|----|------------|-----------|
| O-3 | Бот задаёт вопрос: «Кто вы по роли?» — варианты: деливери-менеджер, скрам-мастер, change-менеджер, агент изменений, другое (свободный ввод). | Must |
| O-4 | Бот задаёт вопрос: «Для кого готовите презентацию?» — варианты: C-level, директора направлений, миддл-менеджмент, смешанная аудитория. | Must |
| O-5 | Бот задаёт вопрос: «Какой тип задачи?» — варианты: «Полный цикл: разобраться с проблемой и подготовить презентацию», «Анализ уже есть, нужна только презентация», «Быстрая проверка анализа и истории». | Must |
| O-6 | Бот запрашивает **цель встречи** (свободный ввод или выбор из списка): одобрение решения, старт пилота, масштабирование, защита результатов квартала/пилота, разбор проблемной команды/процесса, подготовка к стратегсессии. | Must |
| O-6a | Бот задаёт вопрос: **«С кем будет проводиться интервью?»** — пользователь указывает роли (C-level, руководитель направления, тимлид, команда, аналитики, стейкхолдеры и т.п.). Уточняющий вопрос: **«Нужно ли составить отдельные вопросы для каждой роли?»** (да/нет). При «да» — бот в дальнейшем предлагает шаблоны вопросов по выбранным ролям. | Must |
| O-6b | Бот задаёт вопрос о **проблематике и желаемом результате**: **«Что именно хотите выяснить / какую проблему разобрать?»** и **«Что хотите получить в качестве решения / какой итоговый результат нужен от анализа/встречи?»** (свободный ввод или выбор из подсказок: понять причины, предложить план изменений, получить одобрение решений, зафиксировать риски и т.п.). Ответы используются для настройки плана анализа и структуры вопросов. | Must |

### 2.3. Сохранение контекста

| ID | Требование | Приоритет |
|----|------------|-----------|
| O-7 | Ответы онбординга сохраняются в рамках одной «сессии» (один кейс/презентация). Пользователь может продолжить позже (сессия привязана к user_id + session_id). | Must |

## 3. Модуль анализа: контекст и проблематика

### 3.1. Сбор контекста проблемы

| ID | Требование | Приоритет |
|----|------------|-----------|
| A1-1 | Бот запрашивает **область проблемы**: процессы, delivery, discovery, эксперименты/пилоты, оргструктура, качество, сроки, другое. | Must |
| A1-2 | Бот запрашивает **спонсора / инициатора** (кто попросил разобраться): свободный ввод или короткий список типов ролей. | Must |
| A1-3 | Бот запрашивает **основные боли/симптомы**, уже озвученные: сорванные цели, рост lead time, жалобы стейкхолдеров, падение качества, блокировки и т.п. (список + возможность добавить своё). | Must |

### 3.2. Количественные данные

| ID | Требование | Приоритет |
|----|------------|-----------|
| A2-1 | Пользователь может **загрузить или вставить данные**: текст (выгрузки, описание метрик), изображения (скрины дашбордов). Бот принимает фото/документ и текстовые сообщения. | Must |
| A2-2 | Пользователь может указать **какие метрики использует**: lead time, cycle time, WIP, блокировки, конверсия, другие (список + своё). | Should |
| A2-3 | Бот задаёт уточняющие вопросы: текущие значения, тренды, «прострелы» (аномалии). Ответы сохраняются и привязываются к метрикам. | Must |
| A2-4 | Бот предлагает **дополнительные разрезы** для проверки: по командам, по статусам, по типам задач, по блокировкам. Пользователь может отметить «уже посмотрел» или «добавлю позже». | Should |

### 3.3. Качественные данные (интервью, опросы)

| ID | Требование | Приоритет |
|----|------------|-----------|
| A3-1 | Пользователь может описать **с кем проводились интервью/созвоны**: роль (C-level, руководитель, тимлид, команда, аналитик) + краткое описание (без персональных данных при необходимости). | Must |
| A3-2 | Пользователь может ввести **ключевые ответы и цитаты** по интервью (текст). Бот сохраняет их с привязкой к роли/источнику. | Must |
| A3-3 | Бот предлагает **шаблоны блоков вопросов** по ролям (C-level, руководитель направления, тимлид, команда, аналитики). Пользователь может выбрать готовый блок или сформировать свой список вопросов. | Must |
| A3-4 | Бот предлагает **схему агрегации ответов**: по темам, по частоте упоминаний, по ролям. Пользователь может заполнить краткую сводку (или пропустить). | Should |

### 3.4. Дерево анализа и подсветка пробелов

| ID | Требование | Приоритет |
|----|------------|-----------|
| A4-1 | На основе введённых данных бот формирует для пользователя **структурированное «дерево анализа»**: стейкхолдеры/роли, проблемы и боли, подтверждающие/опровергающие данные (метрики, цитаты), гипотезы причин, возможные направления решений. | Must |
| A4-2 | Бот **подсвечивает паттерны**: где несколько источников повторяют одно и то же (на основе введённых цитат/тем). | Should |
| A4-3 | Бот **подсвечивает возможные противоречия**: например, «метрики показывают X, в интервью звучит Y» — на основе введённых данных. | Should |
| A4-4 | Бот **указывает непокрытые зоны**: ключевые роли без интервью, метрики без данных, гипотезы без проверки. Пользователь может отметить «не актуально» или добавить данные позже. | Must |

## 4. Модуль решений и плана

| ID | Требование | Приоритет |
|----|------------|-----------|
| S-1 | Бот запрашивает **варианты решений** (текст). Может предложить форму: «Что делаем? Ожидаемый эффект? Как измерим?» | Must |
| S-2 | Бот запрашивает **план/роадмап**: ключевые шаги, сроки (недели/месяцы/кварталы), кто вовлечён. Структура: список шагов с опциональными сроками и ответственными. | Must |
| S-3 | Бот запрашивает **риски и ограничения** и что с ними планируется делать. Пользователь может добавить несколько рисков. | Must |

## 5. Модуль сторителлинга и структуры презентации

### 5.1. Структура истории

| ID | Требование | Приоритет |
|----|------------|-----------|
| P1-1 | Бот фиксирует **цель презентации и ключевой месседж** (кратко). Может предложить формулировку на основе цели встречи из онбординга. | Must |
| P1-2 | Бот предлагает **структуру истории**: например, контекст → проблема → данные → варианты решений → выбранное решение → план → риски. Альтернатива: SCQA/SCQ&A. Пользователь выбирает или оставляет предложенную. | Must |

### 5.2. Блоки слайдов и черновик текста

| ID | Требование | Приоритет |
|----|------------|-----------|
| P2-1 | Для каждого блока (приветственный слайд, о чём презентация, проблемы, аналитика/данные, решения, роадмап, риски, «спасибо за вопросы») бот задаёт уточняющие вопросы и на основе ответов + собранного анализа формирует **черновой текст**: заголовок слайда (как вывод), 3–5 пунктов. | Must |
| P2-2 | Бот подсказывает **какую 1–2 цифры или диаграмму** стоит показать на слайде (на основе введённых метрик и контекста). | Should |
| P2-3 | Пользователь может **редактировать черновик** по блокам: отправить исправленный текст, бот сохраняет и учитывает при генерации презентации. | Must |
| P2-4 | Бот помогает **упростить формулировки** по запросу: «переформулируй проще» / «короче» для выбранного блока или абзаца. | Should |

### 5.3. Выдача черновика презентации (текст)

| ID | Требование | Приоритет |
|----|------------|-----------|
| P3-1 | После заполнения блоков бот выдаёт **сводный черновик**: список слайдов с заголовками и текстом (в сообщениях или одним файлом .md/.txt). | Must |

---

## 6. Модуль верстки в корпоративном шаблоне

### 6.1. Загрузка шаблона

| ID | Требование | Приоритет |
|----|------------|-----------|
| V-1 | Пользователь может **загрузить файл корпоративного шаблона** в качестве ответа боту. Бот **предлагает загружать файлы в самых популярных форматах** (PPTX, PDF и др., в зависимости от поддерживаемой генерации). Бот сохраняет файл в рамках сессии. | Must |
| V-1a | Перед или после загрузки бот задаёт **уточняющие вопросы о загружаемой копии шаблона**: это корпоративный шаблон для презентаций перед руководством? есть ли в шаблоне отдельные макеты для титульного слайда, слайда с проблемой, слайда с данными/диаграммой, слайда «спасибо за вопросы»? нужно ли использовать только определённые макеты из этого файла? Ответы используются при генерации презентации. | Should |
| V-2 | Пользователь может указать **типы слайдов/макетов** для блоков (например: слайд «проблема» — макет с заголовком и списком; «данные» — макет с заголовком и местом под диаграмму). Вариант: автоматическое сопоставление по умолчанию (блок → типовой макет). | Should |

### 6.2. Генерация презентации

| ID | Требование | Приоритет |
|----|------------|-----------|
| V-3 | Бот **генерирует презентацию**: заполняет слайды шаблона текстом из черновика (заголовки, списки), соблюдает порядок блоков. | Must |
| V-4 | Бот отдаёт пользователю **файл презентации** (PPTX) для скачивания. | Must |
| V-5 | При отсутствии шаблона бот может предложить **собрать презентацию в простом стандартном шаблоне** (минимальная верстка) или только выдать текстовый черновик. | Should |

### 6.3. Ограничения (из интервью)

| ID | Требование | Приоритет |
|----|------------|-----------|
| V-6 | Не требуется извлечение и хранение финансовых метрик или персональных данных из загруженных файлов в аналитических целях; при хранении файлов — минимальный срок и явное информирование пользователя. | Must |

## 7. Виральность и удержание

### 7.1. Шаринг

| ID | Требование | Приоритет |
|----|------------|-----------|
| R-1 | После генерации презентации бот предлагает **готовый короткий текст** для пересылки коллегам с ссылкой на бота (deep link или t.me/...). Пользователь может скопировать и отправить. | Should |
| R-2 | Опционально: **«совместная сессия»** — пользователь получает ссылку на текущий сценарий (read-only или с правом доработки), которой может поделиться с коллегой. Реализация — после MVP при наличии ресурсов. | Could |

### 7.2. Сохранение и возврат

| ID | Требование | Приоритет |
|----|------------|-----------|
| R-3 | Пользователь может **сохранить текущий кейс** (анализ + черновик презентации) под именем и вернуться к нему позже: список «Мои кейсы», выбор кейса, продолжение редактирования или генерация новой версии презентации. | Should |
| R-4 | Бот может предлагать **шаблоны сценариев** под повторяющиеся поводы: «Итоги квартала», «Итоги пилота», «Разбор проблемной команды», «Подготовка к стратегсессии». При выборе шаблона подставляются типовые блоки вопросов и структура слайдов. | Could |

## 8. Нефункциональные требования

### 8.1. Платформа и каналы

| ID | Требование |
|----|-------------|
| NF-1 | Основной канал взаимодействия — **Telegram** (Bot API). |
| NF-2 | Поддержка ввода: текст, фото, документы. **В качестве ответа на вопросы бота пользователь может загружать файлы в самых популярных форматах**: документы (DOC, DOCX, PDF), таблицы (XLS, XLSX), презентации (PPTX), текст (TXT, MD), изображения (JPG, PNG и др.). Ограничения по размеру файлов — по возможностям Telegram API. |

### 8.2. Безопасность и конфиденциальность

| ID | Требование |
|----|-------------|
| NF-3 | Не хранить **финансовые метрики** и данные, позволяющие идентифицировать конкретных людей/команды, дольше чем необходимо для генерации презентации в рамках одной сессии (или обеспечить явное согласие и политику хранения). |
| NF-4 | Загруженные файлы шаблонов и пользовательский контент обрабатывать в рамках сессии; хранение — по минимально необходимому сроку с возможностью очистки. |
| NF-5 | Для корпоративных сценариев предусмотреть возможность работы как **внутреннего решения** (развёртывание у заказчика, без передачи данных во внешний облачный сервис). |

### 8.3. Удобство (UX)

| ID | Требование |
|----|-------------|
| NF-6 | Пользователь в любой момент может **вернуться к предыдущему шагу** или **начать заново** (очистить сессию и стартовать новый кейс). |
| NF-7 | Сообщения бота должны быть **краткими**, с явными вариантами ответа (кнопки/меню) где возможно, чтобы не перегружать чат. |
| NF-8 | При длительной обработке (генерация презентации) — индикация ожидания («Генерирую презентацию…») и обработка таймаутов. |

### 8.4. Кнопка «Нужна помощь»

| ID | Требование | Приоритет |
|----|-------------|-----------|
| NF-9 | **Кнопка «Нужна помощь»** доступна **на всех шагах** диалога (в клавиатуре или меню). При нажатии бот предлагает пользователю **написать в свободной форме**, какая ему нужна помощь. | Must |
| NF-10 | После отправки пользователем текста бот подтверждает приём («Ваш запрос передан, с вами свяжутся») и **отправляет администратору уведомление**: о том, что пользователю нужна помощь, **ник/username пользователя** (Telegram @username или id) и **текст сообщения** пользователя. Канал уведомления администратора: Telegram-чат/группа или личные сообщения (настраиваемый chat_id администратора). | Must |

## 9. Приоритизация для MVP

**Must (MVP):**
- Онбординг: роль, аудитория, тип задачи, цель встречи, с кем интервью и вопросы по ролям, проблематика и желаемый результат (O-1 – O-7, O-6a, O-6b).
- Анализ: контекст проблемы, количественные и качественные данные, дерево анализа, подсветка непокрытых зон (A1-1 – A4-4 в части Must).
- Решения и план (S-1 – S-3).
- Сторителлинг: структура истории, черновик по блокам, сводный черновик (P1-1 – P3-1).
- Верстка: загрузка шаблона, генерация PPTX, выдача файла (V-1 – V-4, V-6).
- Кнопка «Нужна помощь» на всех шагах и уведомление администратору (NF-9, NF-10).
- Базовые нефункциональные требования (NF-1 – NF-4, NF-6 – NF-8).

**Should (ближайшие итерации после MVP):**
- Дополнительные разрезы метрик (A2-2, A2-4), агрегация ответов (A3-4), паттерны и противоречия (A4-2, A4-3).
- Подсказки по визуализации (P2-2), упрощение формулировок (P2-4).
- Маппинг макетов слайдов (V-2), fallback-шаблон (V-5).
- Шаринг с коллегами (R-1), сохранение кейсов (R-3).

**Could (бэклог):**
- Совместные сессии (R-2), шаблоны сценариев (R-4).
- Интеграции (Jira, Confluence, внутренние дашборды) — отдельный документ при появлении запроса.
