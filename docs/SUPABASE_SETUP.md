# Настройка Supabase для хранения данных бота

Бот может сохранять в БД: пользователей, аналитику проблем, диаграммы решений, обратную связь «Нужна помощь». Подключение к Supabase возможно **двумя способами**: через **REST API** (переменные `SUPABASE_URL` и `SUPABASE_SERVICE_ROLE_KEY`) или напрямую к PostgreSQL по `DATABASE_URL`. RAG и скрипт инициализации схемы используют `DATABASE_URL`.

**Настройка Supabase вместе с Railway (одна цепочка шагов):** см. [SUPABASE_AND_RAILWAY.md](SUPABASE_AND_RAILWAY.md).

---

## Подробная инструкция: проект, vector, connection string

### Шаг 1. Создать проект в Supabase

1. **Откройте сайт Supabase**  
   Перейдите в браузере на [https://supabase.com](https://supabase.com).

2. **Войдите или зарегистрируйтесь**  
   Нажмите **Sign in** (или **Start your project**). Вход через GitHub, Google или email. Если аккаунта нет — создайте его.

3. **Создайте организацию (если её ещё нет)**  
   После входа Supabase может предложить создать организацию (Organization). Укажите имя (например, личный аккаунт или название команды) и нажмите **Create organization**.

4. **Создайте новый проект**  
   - Нажмите **New Project** (кнопка справа вверху или на главной дашборда).  
   - Выберите **организацию**, в которой будет проект (если их несколько).  
   - **Name**: введите имя проекта, например `preza_bot` или `preza-bot`. Имя будет в URL дашборда.  
   - **Database Password**: придумайте и введите надёжный пароль для пользователя БД `postgres`.  
     - **Обязательно сохраните пароль** в менеджере паролей или в надёжном месте — он понадобится для connection string и без него подключиться к БД будет нельзя.  
     - Пароль можно сбросить позже в Project Settings → Database, но тогда придётся обновить все места, где используется старый пароль.  
   - **Region**: выберите регион сервера (например, `Frankfurt (eu-central-1)` или ближе к вашим пользователям / к Railway). От региона зависит задержка при обращении к БД.  
   - **Pricing Plan**: выберите Free, если достаточно бесплатного тарифа.  
   - Нажмите **Create new project**.

5. **Дождитесь создания проекта**  
   Создание занимает 1–2 минуты. На экране будет индикатор прогресса. Когда проект готов, откроется дашборд проекта (главная с меню слева: Table Editor, SQL Editor, Database и т.д.).

### Шаг 2. Включить расширение vector (pgvector)

Расширение **vector** нужно для RAG (база знаний с эмбеддингами) и для работы бота с pgvector.

1. **Откройте настройки проекта**  
   В левом меню внизу нажмите на иконку **шестерёнки** (Settings). Откроется раздел настроек.

2. **Перейдите в настройки Database**  
   В боковом меню настроек выберите **Database**.

3. **Откройте раздел Extensions**  
   На странице Database прокрутите вниз до блока **Extensions** (или найдите его в подменю, если интерфейс изменился).

4. **Включите vector**  
   - В списке расширений найдите **vector** (может быть подписан как «pgvector» или «vector (pgvector)»).  
   - Нажмите переключатель или кнопку **Enable** рядом с **vector**.  
   - Дождитесь подтверждения (иконка галочки или статус «Enabled»).  
   - Если **vector** нет в списке — убедитесь, что вы в правильном проекте и что регион поддерживает расширения; при необходимости проверьте [документацию Supabase](https://supabase.com/docs/guides/database/extensions).

После включения расширение активно для всей БД проекта; отдельно настраивать его для таблиц не нужно.

### Шаг 3. Получить connection string (URI)

Connection string — это строка подключения к PostgreSQL, которую бот и скрипты используют в переменной `DATABASE_URL`.

1. **Остаёмся в настройках Database**  
   В левом меню: **Settings** (шестерёнка) → **Database**. Если вы ушли со страницы — вернитесь так же.

2. **Найдите блок Connection string**  
   На странице Database найдите секцию **Connection string** (или **Connection info**). В ней обычно несколько вкладок: **URI**, **JDBC**, **DotNet** и т.д.

3. **Выберите вкладку URI**  
   Переключитесь на вкладку **URI**. В поле будет строка вида:
   ```text
   postgresql://postgres.[PROJECT-REF]:[YOUR-PASSWORD]@aws-0-[REGION].pooler.supabase.com:6543/postgres
   ```
   Здесь `[PROJECT-REF]` — идентификатор проекта, `[REGION]` — код региона, `[YOUR-PASSWORD]` — плейсхолдер для пароля.

4. **Подставьте свой пароль**  
   - Скопируйте всю строку (кнопка **Copy** рядом с полем или выделение вручную).  
   - Вставьте в текстовый редактор.  
   - Замените **`[YOUR-PASSWORD]`** на пароль базы данных, который вы задали при создании проекта (шаг 1.4).  
   - Если в пароле есть спецсимволы (`@`, `#`, `%`, `/` и т.д.), их нужно **закодировать в URL** (percent-encoding). Например: `@` → `%40`, `#` → `%23`. Или задайте пароль без спецсимволов при создании проекта.  
   - Пробелы в пароле не допускаются; используйте пароль без пробелов.

5. **Какой режим пула использовать (порты 6543 и 5432)**  
   - **Порт 6543** (Connection pooler, Transaction mode) — строка обычно уже с портом `6543` и хостом `pooler.supabase.com`. Подходит для серверных приложений (в т.ч. бот на Railway).  
   - **Порт 5432** (Direct connection) — для прямого подключения к БД без пула. Нужен, например, для миграций или если с пулером возникают проблемы с pgvector.  
   - Для бота и скрипта `init_supabase_schema` чаще всего достаточно URI с портом **6543**. Если при работе RAG или расширении vector появятся ошибки, попробуйте строку с портом **5432** (в разделе Connection string часто есть переключатель «Session mode» / «Direct» или отдельная строка для Direct connection).

6. **Session mode и pgvector (если используете RAG)**  
   При использовании **Connection pooler** (порт 6543) в конце URI может быть параметр `?pgbouncer=true`. Для запросов с pgvector иногда нужен **Session mode**: замените в URI `?pgbouncer=true` на `?pgbouncer=false` (или добавьте `?pgbouncer=false`, если параметров ещё не было). Тогда пулер будет в режиме сессии и расширение vector будет работать стабильно.

7. **Итоговая строка — это ваш DATABASE_URL**  
   Финальная строка без плейсхолдеров — это значение для переменной окружения **DATABASE_URL**. Пример (пароль и ref/region подставьте свои):
   ```text
   postgresql://postgres.xxxxxxxxxxxx:MySecretPass123@aws-0-eu-central-1.pooler.supabase.com:6543/postgres?pgbouncer=false
   ```
   - Эту строку **не коммитить в репозиторий** и не светить в логах.  
   - Локально: положите в `.env`: `DATABASE_URL=postgresql://...`.  
   - На Railway: в настройках сервиса бота → **Variables** → добавьте переменную **DATABASE_URL** со значением этой строки.

## Шаг 4. Инициализировать схему один раз

Локально (с `DATABASE_URL` в `.env`):

```bash
python -m scripts.init_supabase_schema
```

Или в Railway: одноразовый run (если доступен) или локально с `DATABASE_URL` из переменных Railway.

Скрипт создаёт таблицы: `telegram_users`, `analyses`, `diagrams`, `feedback_requests`, включает расширение `vector`.

## Шаг 5. Переменные окружения

**Вариант A — сохранение данных через REST API (рекомендуется для Railway):**

- **SUPABASE_URL** — Project URL из Supabase: **Settings** → **API** → Project URL (например `https://xxxx.supabase.co`).
- **SUPABASE_SERVICE_ROLE_KEY** — **Settings** → **API** → Project API keys → **service_role** (secret). Не светить и не коммитить.
- При заданных обоих бот пишет в таблицы через Supabase REST API; **DATABASE_URL** для сохранения не обязателен (нужен только для RAG и для запуска `scripts.init_supabase_schema`).

**Вариант B — прямое подключение к PostgreSQL:**

- **DATABASE_URL** — URI из шага 3 (обязательно для сохранения и RAG при этом варианте).

**Общее:**

- **PERSIST_TO_DB** — `true` чтобы бот писал в таблицы (по умолчанию `true`, если задан хотя бы один способ: REST или DATABASE_URL).

В Railway добавьте выбранные переменные в **Variables**. Секреты не коммитить.
