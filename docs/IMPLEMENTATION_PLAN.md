# План реализации бота в Telegram (для новичка в разработке)

План основан на **описании продукта** (`docs/product_descriptions/`), **функциональных требованиях** (`docs/BOT_REQUIREMENTS.md`) и **сценарии диалога** (`docs/BOT_FLOW.md`). Каждый этап можно делать по очереди и проверять в боте.

---

## Что понадобится перед стартом

- Установленный **Python 3.10+**
- Аккаунт в **Telegram** и токен бота от [@BotFather](https://t.me/BotFather)
- Редактор кода (Cursor, VS Code и т.п.)
- Базовое понимание: **функции**, **словари/списки**, **async/await** (можно подтянуть по ходу)

**Полезные ссылки для новичка:**
- [python-telegram-bot: Tutorial](https://docs.python-telegram-bot.org/en/stable/tutorial.html) — как устроен бот
- [Telegram Bot API](https://core.telegram.org/bots/api) — что умеют боты

## Этап 0. Подготовка окружения и «пустой» бот

**Цель:** запустить бота так, чтобы на `/start` он что-то отвечал.

### Шаги

1. **Клонировать/открыть проект**, перейти в корень (где лежат `run.py`, `bot/`).
2. **Создать виртуальное окружение** (в терминале):
   ```bash
   python3 -m venv venv
   source venv/bin/activate   # Windows: venv\Scripts\activate
   ```
3. **Установить зависимости:**
   ```bash
   pip install -r requirements.txt
   ```
4. **Создать файл `.env`** (скопировать из `.env.example`), прописать `BOT_TOKEN` от BotFather.
5. **Запустить бота:**
   ```bash
   python run.py
   ```
6. **Проверить:** написать боту в Telegram `/start` — должен прийти ответ (сейчас приветствие из `bot/handlers/commands.py`).

**Где смотреть код:** `bot/main.py` (регистрация обработчиков), `bot/handlers/commands.py` (обработка `/start`, `/help`).

**Критерий готовности:** бот запускается без ошибок и отвечает на `/start`.

## Этап 1. Состояние диалога (state) и кнопки

**Цель:** хранить «на каком шаге пользователь» и показывать кнопки (Reply Keyboard).

### Шаги

1. **Решить, где хранить состояние.** Для начала достаточно **в памяти** (словарь `user_id → состояние`). Позже можно перенести в Redis/БД.
   - Пример структуры: `user_states[user_id] = {"step": "onboarding_role", "scenario": None, "onboarding": {}}`.
2. **Создать модуль состояний**, например `bot/storage/session.py`:
   - функция `get_state(user_id)` — вернуть текущее состояние или дефолт;
   - функция `set_state(user_id, state)` — сохранить состояние;
   - функция `clear_state(user_id)` — сбросить (для «Начать сначала»).
3. **Подключить кнопки** (ReplyKeyboardMarkup из `telegram`):
   - в `commands.py` в обработчике `/start` показывать кнопки: «Помощь с аналитикой», «Вопросы для интервью», «Подготовить презентацию», «Полный цикл»;
   - при нажатии на кнопку пользователь отправит текстовое сообщение — его будет обрабатывать `messages.handle_message`.
4. **В `handle_message`** по тексту сообщения и текущему `state` решать, что делать дальше (переход на следующий шаг, сохранение ответа в состояние).

**Документация:** [ReplyKeyboardMarkup](https://docs.python-telegram-bot.org/en/stable/telegram.replykeyboardmarkup.html).

**Критерий готовности:** после `/start` видны 4 кнопки; при нажатии одной из них состояние сохраняется (можно залогировать или вывести следующим сообщением).

## Этап 2. Онбординг (шаги 2.1–2.7)

**Цель:** пройти все вопросы онбординга по сценарию из `BOT_FLOW.md` (роль, аудитория, тип задачи, цель встречи, с кем интервью, нужны ли вопросы по ролям, проблематика и желаемый результат).

### Шаги

1. **Завести в состоянии поля** под онбординг: `onboarding.role`, `onboarding.audience`, `onboarding.task_type`, `onboarding.meeting_goal`, `onboarding.interview_roles`, `onboarding.questions_per_role`, `onboarding.problem_and_result`.
2. **Реализовать пошаговый сценарий:**
   - после выбора сценария (одна из 4 кнопок) перейти к шагу 2.1;
   - на каждом шаге: отправить вопрос + кнопки (если есть фиксированные варианты) или запрос текста;
   - в `handle_message` по `state.step` понять, какой ответ пришёл, сохранить в `onboarding`, перейти к следующему шагу.
3. **Тексты вопросов** взять из `BOT_FLOW.md` (Шаг 2).
4. **После 2.7** сделать ветвление по `scenario`: «Вопросы для интервью» → шаг 4; «Подготовить презентацию» → шаг 8 (или 7); «Помощь с аналитикой» / «Полный цикл» → шаг 3.

**Критерий готовности:** можно пройти все 7 вопросов онбординга, ответы сохраняются, после последнего вопроса бот переводит на следующий блок по выбранному сценарию.

## Этап 3. Кнопка «Нужна помощь» и уведомление администратору

**Цель:** на каждом шаге показывать кнопку «Нужна помощь»; по нажатию — запрос текста, отправка уведомления админу, возврат в диалог или в меню.

### Шаги

1. **Добавить кнопку «Нужна помощь»** во все клавиатуры (общая функция для построения клавиатуры: «основные кнопки шага» + «Нужна помощь»).
2. **Обработка нажатия «Нужна помощь»:**
   - сохранить в состоянии `return_after_help` текущий шаг (чтобы вернуться);
   - перевести в шаг 0H.1: отправить «Опишите, какая вам нужна помощь (в свободной форме)».
3. **Когда пользователь пришлёт текст в состоянии 0H.1:**
   - ответить: «Ваш запрос передан. С вами свяжутся.»;
   - отправить администратору сообщение: текст вида «Пользователю нужна помощь», ник/username (или `user_id`), текст сообщения пользователя;
   - chat_id администратора хранить в настройках (например `ADMIN_CHAT_ID` в `.env`).
4. **Показать кнопки:** «Вернуться в диалог» / «В главное меню»; по нажатию — восстановить шаг из `return_after_help` или перейти в шаг 1.

**Критерий готовности:** с любого шага по «Нужна помощь» можно написать текст, админ получает уведомление с ником и текстом, после этого можно вернуться в диалог или в меню.

## Этап 4. Контекст проблемы (шаг 3)

**Цель:** собрать область проблемы, спонсора, боли/симптомы (только для сценариев «Помощь с аналитикой» и «Полный цикл»).

### Шаги

1. **В состоянии добавить** `context`: `area`, `sponsor`, `pains`.
2. **Задать три вопроса по очереди** (тексты из `BOT_FLOW.md`, шаг 3):
   - область: кнопки (процессы, delivery, discovery, эксперименты/пилоты, оргструктура, качество, сроки, другое);
   - спонсор: текст или кнопки;
   - боли/симптомы: множественный выбор или текст.
3. **Сохранять ответы** в `context`, переходить к следующему вопросу или к шагу 4/5 по сценарию.

**Критерий готовности:** после онбординга при сценарии «Полный цикл» бот задаёт три вопроса контекста и переходит к вопросам по ролям или к сбору данных.

## Этап 5. Вопросы для интервью по ролям (шаг 4)

**Цель:** показать шаблоны вопросов по выбранным ролям и выдать итоговый набор (для сценария «Вопросы для интервью» или при «Полный цикл» + нужны вопросы по ролям).

### Шаги

1. **Подготовить контент:** шаблоны блоков вопросов по ролям (C-level, руководитель, тимлид, команда, аналитики). Хранить в коде или в файлах (например, `bot/data/interview_templates.json`).
2. **По выбранным в онбординге ролям** показать соответствующие блоки вопросов (текстом или кнопками «Использовать» / «Редактировать»).
3. **Собрать итоговый список** и отправить пользователю (одним сообщением или файлом).
4. **Предложить:** «Перейти к сбору ответов (анализ)?» или «В главное меню».

**Критерий готовности:** после выбора «Вопросы для интервью» и прохождения онбординга пользователь получает набор вопросов по ролям; при «Полный цикл» с «да» на вопросы по ролям — то же самое, затем переход к данным.

## Этап 6. Количественные и качественные данные (шаг 5)

**Цель:** приём текста и файлов (метрики, скрины, описание интервью, цитаты), сохранение в состоянии.

### Шаги

1. **Приём файлов:** в `main.py` добавить обработчик для `MessageHandler(filters.PHOTO | filters.Document, ...)`. В обработчике по текущему шагу понять, что пользователь прислал «данные», сохранить `file_id` или текст в `quantitative_data` / `qualitative_data`.
2. **Пошаговые подсказки:** «Загрузите или вставьте данные…», «Какие метрики ключевые?», «Текущие значения и тренды?», «С кем проводились интервью?», «Ключевые ответы и цитаты?» (см. `BOT_FLOW.md`, шаг 5).
3. **Сохранять** в состояние списки/тексты; дать возможность «пропустить» или «добавлю позже» кнопками.
4. **Переход к шагу 6** — когда минимум данных собран или пользователь нажал «Дальше».

**Критерий готовности:** пользователь может отправить текст и файлы, бот задаёт уточняющие вопросы и переходит к дереву анализа.

## Этап 7. Дерево анализа и непокрытые зоны (шаг 6)

**Цель:** сформировать из собранных данных структурированное «дерево» и подсветить непокрытые зоны.

### Шаги

1. **Собрать из состояния** контекст, количественные и качественные данные.
2. **Сформировать текст «дерева анализа»:** стейкхолдеры/роли, проблемы и боли, подтверждающие данные, гипотезы причин, возможные решения (шаблон можно захардкодить или собирать из полей).
3. **Непокрытые зоны:** сравнить выбранные роли для интервью с теми, по ком есть данные; метрики без значений; вывести списком «возможно, стоит добавить: …».
4. **Отправить пользователю** дерево + непокрытые зоны, кнопки «Добавлю позже» / «Дальше».

**Критерий готовности:** после ввода данных бот выдаёт структурированное дерево и список непокрытых зон и переходит к решениям и плану.

## Этап 8. Решения и план (шаг 7)

**Цель:** запросить варианты решений, роадмап, риски.

### Шаги

1. **Три блока вопросов** (тексты из требований S-1, S-2, S-3):
   - варианты решений (что делаем, эффект, как измерим);
   - план/роадмап (шаги, сроки, кто вовлечён);
   - риски и ограничения, что с ними делать.
2. **Сохранять** в состояние `solutions`.
3. **Переход к шагу 8** (сторителлинг).

**Критерий готовности:** бот задаёт три блока, сохраняет ответы и переходит к структуре презентации.

## Этап 9. Сторителлинг и черновик слайдов (шаг 8)

**Цель:** зафиксировать цель и месседж, предложить структуру истории, для каждого блока слайдов запросить/сгенерировать черновой текст и выдать сводный черновик.

### Шаги

1. **Цель и месседж** — один вопрос, можно подсказать из `onboarding.meeting_goal`.
2. **Структура истории** — кнопки (например: «Контекст → проблема → данные → решения → план → риски» или SCQA).
3. **По каждому блоку слайдов** (приветственный, о чём презентация, проблемы, данные, решения, роадмап, риски, «спасибо за вопросы»):
   - задать уточняющий вопрос или подставить черновик из собранных данных;
   - принять правки от пользователя (текст сообщения);
   - сохранить в `story_blocks`.
4. **Сводный черновик** — сформировать один текст (или файл .md/.txt) со всеми слайдами и отправить. Кнопка «Перейти к сборке презентации».

**Критерий готовности:** пользователь получает пошагово черновики блоков и в конце — сводный черновик, после чего переходит к загрузке шаблона.

## Этап 10. Загрузка шаблона и генерация PPTX (шаги 9–10)

**Цель:** принять файл шаблона (PPTX), при необходимости задать уточняющие вопросы о макетах, сгенерировать презентацию и отдать файл.

### Шаги

1. **Запрос шаблона:** «Загрузите корпоративный шаблон (PPTX, PDF и т.д.)». Обработчик документов сохраняет `file_id` в состояние.
2. **Уточняющие вопросы (V-1a):** про макеты (титульный, проблема, данные, «спасибо»), использовать ли только определённые — ответы сохранить.
3. **Генерация PPTX:** использовать библиотеку (например, `python-pptx`) или внешний сервис: взять текст из `story_blocks`, заполнить слайды шаблона. Если шаблон не загружен — выдать только черновик (текст) или предложить стандартный шаблон.
4. **Отправка файла:** сгенерированный PPTX отправить пользователю через `context.bot.send_document(chat_id=..., document=...)`.
5. **Обработка ошибок:** при таймауте или ошибке — сообщение и кнопки «Повторить» / «Только черновик».

**Критерий готовности:** пользователь загружает шаблон, бот генерирует и отправляет PPTX (или черновик при ошибке).

## Этап 11. После генерации: шаринг и сохранение кейса (шаг 11)

**Цель:** предложить текст для шаринга с ссылкой на бота, опционально сохранить кейс под именем.

### Шаги

1. **Текст для шаринга** — готовое сообщение со ссылкой на бота (t.me/YourBot).
2. **«Сохранить кейс?»** — запросить имя; сохранить в хранилище (пока можно файл или словарь в памяти; позже — БД).
3. **Кнопки** «Начать новый кейс» / «В главное меню» — сброс состояния и переход к шагу 1.

**Критерий готовности:** после выдачи файла пользователь видит предложение поделиться и сохранить кейс и может начать заново.

## Этап 12. Команды «Назад» и «Справка»

**Цель:** на любом шаге можно вернуться на шаг назад или в начало; справка всегда доступна.

### Шаги

1. **Кнопка «Назад»** (где применимо): по нажатию восстановить предыдущий шаг из истории или из карты переходов.
2. **Команда /help:** краткий список команд и описание; кнопка «В начало» → сброс состояния, переход к шагу 1.
3. **«Начать сначала»** в меню — то же, что сброс и шаг 1.

**Критерий готовности:** пользователь может откатиться на шаг назад и в любой момент вызвать справку или начать сначала.

## Порядок реализации (рекомендуемый для новичка)

1. **Этап 0** — запуск бота.  
2. **Этап 1** — состояние и кнопки.  
3. **Этап 2** — онбординг целиком.  
4. **Этап 3** — «Нужна помощь» и админ.  
5. **Этапы 4–7** — контекст, вопросы по ролям, данные, дерево анализа.  
6. **Этапы 8–9** — решения, план, сторителлинг и черновик.  
7. **Этап 10** — шаблон и генерация PPTX (самый трудоёмкий).  
8. **Этапы 11–12** — шаринг, сохранение, «Назад», справка.

После каждого этапа проверять сценарий в Telegram.

## Уточняющие вопросы перед/во время разработки

Ниже вопросы, ответы на которые упростят реализацию и приоритизацию.

### Продукт и сценарии

1. **MVP:** делать сразу все четыре сценария («Помощь с аналитикой», «Вопросы для интервью», «Подготовить презентацию», «Полный цикл») или сначала один (например, «Полный цикл»)?
2. **«Подготовить презентацию» (анализ уже есть):** сколько вопросов задаём перед черновиком? Только цель встречи и краткий контекст или мини-онбординг?
3. **Шаблоны вопросов для интервью:** у вас уже есть готовые тексты по ролям (C-level, тимлид и т.д.) или их нужно придумать в рамках проекта?

### Техника

4. **Хранение состояния:** до MVP достаточно хранения в памяти (словарь по `user_id`) или сразу нужна БД/Redis (чтобы при перезапуске бота сессии не терялись)?
5. **Генерация PPTX:** планируется только заполнение текстом существующих слайдов шаблона или ещё разметка/диаграммы из данных? Есть ли предпочтение по библиотеке (python-pptx и т.п.)?
6. **Администратор:** один chat_id в `.env` достаточно или нужна поддержка нескольких админов (например, группа)?

### Контент и тексты

7. **Тексты сообщений бота:** править их будете вы (в коде/конфиге) или нужен вынос в файлы (например, YAML/JSON) для нетехнического редактора?
8. **Fallback-шаблон:** если пользователь не загрузит шаблон, выдаём только текстовый черновик или нужен встроенный простой PPTX-шаблон «по умолчанию»?

Ответы на эти вопросы можно добавить в этот документ или в `BOT_REQUIREMENTS.md` и ориентироваться на них при реализации.
